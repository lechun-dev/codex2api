# Changelog

## v2.6.0 - 2026-07-23

### Features

- **Configurable, auditable, gradual-rollout layered Prompt Guard for the V1 APIs (#429, by @ifThink404).** An opt-in gateway-side protection layer inspects V1 requests across `/v1/responses`, `/v1/chat/completions`, `/v1/messages`, the Responses WebSocket, and a new text Realtime WebSocket, to reduce the risk of shared upstream accounts being flagged by the provider's safety systems for jailbreak / prompt-injection traffic. It splits the request into independent origins — current-user prompt, history, system/developer instructions, tool output, tool arguments, attachments, session context — so tool output or app context can't inflate the user's execution score, and separates an **execution score** (the only one that can block/strike) from a **shadow audit score** (observe-only). Detection adds bounded normalization/decoding (URL, HTML, Base64, Hex, ROT13, escapes, compression) under a scan budget, combination rules with evidence binding, strict terminal rules, per-provider profiles (balanced/strict/research), user/IP/session cumulative risk, per-user/-key percentage rollout, an optional semantic sidecar and attachment/intelligence extension points (each independently switchable with timeout/concurrency/cache/circuit-breaker), and SSE/WebSocket output scanning before frames reach the downstream. A signed **NewAPI audit protocol** passes through and verifies user id / real IP / request id / timestamp / HMAC and returns structured strike/ban evidence, backed by an async audit queue. The admin **Prompt Filter** page gains full visual configuration, rule intelligence history/diffs, trigger logs and complete audit context. Every switch — the master toggle, NewAPI policy, strikes, account bans, IP blacklist, and the auxiliary layers — **defaults to off / monitor**, and the recommended path is the documented four-stage grayscale (monitor → warn/small-traffic → targeted enforce → strict terminal + auto-strike). One operational caveat is documented in-app: with output scanning on, `codex_preflight_sse_passthrough` no longer reliably restores original first-frame speed, so the two must be grayscaled together.
- **Grok upstream hardening — cooldown persistence, periodic probe, retry budget, and model registration.** Grok accounts persist per-model cooldowns across restarts, run a configurable background connectivity probe (`grok_config`, default off / 30 min) that rewrites account status, gain a per-account rate-limit retry budget (`grok_config.max_rate_limit_retries`, `0` = follow global), clamp reasoning effort to Grok's supported band (`xhigh`/`max` → `high`, `minimal` → `low`) on the mandatory sanitize path, and — when they declare no model whitelist — auto-register a default Grok model set (`grok-4.5`, `grok-4`, `grok-3-fast`, …) into `/v1/models`. Free-tier `429` responses now surface authoritative token usage and a rolling-24h quota bar instead of a blind 30-minute cooldown.
- **Generic API keys can schedule undeclared Grok accounts via the default model set.** A key with no upstream-channel pin previously could never reach a Grok account that declared no model whitelist (the relay filter treated an empty whitelist as "serves nothing"), so `grok-4.5` returned a permanent 503 even though `/v1/models` advertised it. Undeclared Grok accounts now serve exactly `DefaultGrokModelIDs`, matching what the model list advertises, while non-Grok models still can't leak onto Grok accounts.
- **Proxy-pool selector in the account forms, and OAuth Grok accounts hide the fixed Base URL.** Both the Codex and Grok account add/edit forms grow a "select from proxy pool" dropdown that fills the chosen proxy's URL into the still-editable field (hidden when the pool is empty). OAuth Grok accounts use the official `cli-chat-proxy` endpoint, so the Base URL field now shows only for API Key accounts in both the add and edit dialogs.
- **`/v1/messages` thinking round-trip — reasoning signatures replayed as `encrypted_content`.** On the Anthropic endpoint, the translator now emits a `signature_delta` carrying the upstream reasoning `encrypted_content`, and on the input path rebuilds signed thinking blocks into `reasoning` items, so a multi-turn Claude Code conversation keeps the upstream's encrypted reasoning context instead of dropping it every turn. Verified end to end on both Codex (`gpt-5.5`) and Grok (`grok-4.5`) upstreams.
- **Optional preflight SSE passthrough compatibility switch (#425).** A default-off, hot-reloadable `codex_preflight_sse_passthrough_enabled` restores the legacy behavior of forwarding pre-content metadata events (`codex.rate_limits` / `codex.response.metadata` / `response.metadata`) immediately instead of buffering them until first content — for downstream gateways that treat the first SSE data frame as time-to-first-response. The admin UI warns that enabling it commits HTTP 200 before first content, reverting the pre-content-failure recovery semantics.
- **Optional `first_token_ms` accounting that excludes WS connection-acquire time.** A new setting lets the logged `first_token_ms` subtract the per-attempt WS connection-acquire cost (default off, keeping the original acquire-inclusive figure); the raw value is still `first_token_ms + ws_acquire_ms`.
- **Grok-only bulk clean (banned/error) with custom confirm dialogs.** The Grok accounts page gets scoped clean actions and a plan filter, with dedicated confirm dialogs.
- **OAuth accounts dedupe by workspace identity — `email + workspace_id` (#431, by @jenfonro).** OAuth dedup previously keyed on `email + account_id/chatgpt_account_id/user_id`, which some import sources and legacy backfill polluted with `user-*` / legacy shapes, so accounts under one email couldn't be reliably de-duplicated across (or within) workspaces. Standard Codex OAuth accounts now dedupe on an independent `workspace_id`, parsed from the JWT only when the same token yields both `email` and `chatgpt_account_id`: imports, OAuth authorization, self-service submission, and account restore all match on `email + workspace_id`; RT/session refresh writes the workspace identity back into credentials to backfill bare-RT / legacy accounts; and a one-time `20260722_workspace_identity_v3` data migration backfills missing `workspace_id` from the JWT (active-pool accounts only, only when the JWT email matches, never touching the recycle bin or legacy `account_id`) and merges the resulting `email + workspace_id` duplicates. When `workspace_id` is known it now takes precedence over the `allow_duplicate` flag — an account explicitly imported as a duplicate is still merged if it provably shares a workspace; `allow_duplicate` only preserves duplicates when the workspace can't be identified. Agent Identity, Grok, and OpenAI-Responses accounts are excluded from the workspace migration.

### Fixes

- **Credentials-wrapped Agent Identity `auth.json` now imports (#424).** Account exports from sub2api / codex2api store the Agent Identity fields flat inside a `credentials` object rather than under an `agent_identity` sub-object, so the import dialog rejected them with "not Agent Identity format" even though every field was present. The parser now unwraps `credentials` before detection (covering the single paste and the batch file upload), and the generic accounts.json importer synthesizes the identity node from flat credentials instead of silently dropping the entry.
- **Grok account proxy edits persist to the database.** `UpdateGrokAccount` only wrote the credentials JSON, but `proxy_url` is a dedicated column — so editing a Grok account's proxy only patched the in-memory store and silently reverted on reload/restart. The edit now persists `proxy_url` (empty clears it).
- **Quota is charged even when usage logging is disabled (#428).** Turning off usage logs no longer skips quota accounting.
- **Streamed tool calls are cached for continuation (#427).** Tool-call output items from a streamed response are now cached so `previous_response_id` continuation replays them correctly.
- **A deleted Codex Agent runtime marks the account banned (#426).** When the upstream reports the agent runtime no longer exists, the Agent Identity account is banned rather than retried into repeated failures.
- **Pooled WebSocket connections survive upstream trailing metadata frames.** A `responsesapi.websocket_timing` trailing frame no longer kills the pooled connection every turn (the first-token regression root cause); idle-frame adjudication keeps the connection alive.
- **Prompt Filter trigger-log badges no longer overflow into the endpoint column.** In the fixed-layout trigger log, a long origin label ("current user prompt") rendered at its natural width and overlapped the adjacent model text; the action-column badges now wrap within their cell.
- **Prompt Guard background database work drains safely on shutdown (#430, by @ifThink404).** The async audit / probe / intelligence tasks introduced with the prompt guard are now lifecycle-managed: on shutdown the database stops accepting new detached work, waits out a grace period, then cancels and waits for every registered task to exit before closing the connection, so no background writer can outlive the SQL connection. This closes the async-write and temp-SQLite-cleanup races that surfaced as flaky CI after #429, and also brings account import/refresh probes, Grok usage/status probes, rule intelligence, CLI-version sync, and model-manifest learning under the same drain discipline.

## v2.5.9 - 2026-07-22

### Features

- **Grok is now a first-class upstream provider (accounts, scheduling, billing, per-key routing).** Alongside Codex/OpenAI, the pool can hold **Grok** accounts and route `/v1/responses`-shaped traffic to xAI. Accounts import three ways — Device Code OAuth (RFC 8628, the same flow the official Grok CLI uses), `auth.json` paste, and raw `xai-…` API key — plus batch file/`sso.txt`/`refreshtoken.txt` importers and an **SSO→Build converter** that drives the xAI device flow with a `grok.com` `sso` cookie to mint an OAuth account with no manual authorization. A dedicated `/accounts/grok` page (table + card layouts) shows a full billing view (per-product usage, on-demand cap, monthly spend). API keys gain a first-class **upstream channel** field (`auto`/`codex`/`grok`) that hard-gates which account pool serves the key; the Dashboard and Usage pages get an All/Codex/Grok segmented filter (backed by a new `usage_logs.channel` column + backfill migration), and the model dropdowns follow the selected channel (admin `/models` now returns `grok_models`, the union of Grok accounts' declared models). Grok accounts persist their weekly/monthly usage across restarts, carry their own error/cooldown mapping (free-quota exhaustion, spend limits, `Retry-After`), and never run the ChatGPT wham/`/responses` probes that would mis-ban them.
- **Codex Agent Identity auth import — sign-per-request accounts with no stored OAuth token.** A new "Agent Identity" import method accepts a Codex `auth.json` in `auth_mode=agentIdentity` form: the account stores only an Ed25519 (PKCS#8) agent private key + `agent_runtime_id`, never an access/refresh token, and every upstream request is dynamically signed. The gateway registers a task with `auth.openai.com/api/accounts/…/task/register` (decrypting the returned task id via X25519/NaCl anonymous box when the endpoint encrypts it), caches and persists the task id, sends `Authorization: AgentAssertion <signed-envelope>` in place of the bearer token, and re-registers + retries once on a `401 invalid_task_id`. These accounts run over HTTP (their per-request signing model doesn't fit the long-lived WS handshake), show a blue "Agent Identity" badge in the account list, and populate their usage bars after import via a signed `/responses` probe. Import works as a single paste, a multi-file batch, or by auto-detection inside the general CLIProxyAPI/Sub2Api JSON import and folder scan (dedup by `agent_runtime_id`); `auth_mode` is accepted in both `agent_identity` and `agentIdentity` spellings.
- **Context-window overflow auto-compaction (#415).** When an upstream turn overflows the model's context window, the gateway can now transparently compact the conversation and retry instead of surfacing the error — an experimental global setting with a per-key switch and a configurable summary retry, including a fix for the loose-mode gap (#310) and the pre-token frame-buffering pitfall.
- **WebSocket busy-session overflow to sibling connections (#413).** A busy WS session now escalates through a tunable ladder — patience wait → spill to a sibling connection → concurrency cap — instead of stalling, gated behind an env valve (default off) and wired through the full settings pipeline and UI; a new `ws_acquire_ms` segment records connection-acquire time in usage logs.
- **wham credits balance and applicable reset credits surfaced.** The wham usage probe now threads the account's `credits.balance` (previously dropped by `ApplyWhamUsage`) and the `applicable_available_count` reset-credit count through to the account view.

### Fixes

- **The `/v1/messages` Claude→Codex model mapping keys now appear in `/v1/models`.** The global Claude model mapping (used to translate `claude-*` model names to Codex models on the Anthropic endpoint) was never added to the served model list, so pulling `/v1/models` returned no `claude-*` entries and downstream clients couldn't discover them. Every non-wildcard mapping "from" key is now included in the model list.
- **Tool call / output `call_id` pairing repaired on the input path (#414).** Clients that trim conversation history (e.g. Copilot) could drop a `function_call` while keeping its output, producing an unpaired `call_id` that the upstream rejected with 400. A bidirectional, stateless repair reconciles the pairing before dispatch.
- **`/v1/messages` alias-injected `reasoning_effort` stripped, and pre-token upstream failures no longer masked (#412).** `applyMessagesModelMapping` now removes the top-level `reasoning_effort` field the alias injection wrote (it 400s on the Responses upstream), and a failure before the first token is surfaced with its real status code instead of being swallowed by the retry path.
- **Large-body input validation no longer O(N²) (#417).** The prompt filter early-exits when disabled and function-name validation is now lazy; input validation on large request bodies is ~5× faster on the benchmark.
- **Bump `golang.org/x/text` to v0.39.0 (GO-2026-5970).** Fixes a reachable "infinite loop on invalid input" advisory in the previously-pinned v0.38.0, called via the uTLS HTTP/2 transport and the prompt-filter text normalization.

## v2.5.8 - 2026-07-19

### Features

- **Per-key image-tool policy gains a third mode: strip-and-forward (#411).** Beyond the existing allow (default) and block (403 on any image-generation touch), an API key can now be set to **strip**: the gateway removes the Codex image-capability declarations from a `/v1/responses` request and forwards it as a plain text request instead of rejecting it. Stripping covers the flat `{"type":"image_generation"}` tool and the namespace form `{"type":"namespace","name":"image_gen"}` in the top-level `tools[]`, the Responses Lite embedded `input[].additional_tools.tools[]` (an emptied carrier item is removed entirely), a `tool_choice` explicitly pointing at `image_generation`/`image_gen` (`"auto"` and other selections are untouched), and the gateway's own auto-injected image tool + bridge instructions (the strip runs after body preparation, so injection is naturally suppressed). All other tools — shell, function, apply_patch, MCP, web search — survive. This serves upstreams (relay channels, account groups, plans) that reject any request merely *declaring* image capability with `403 Image generation is not enabled for this group`, even when the turn is pure text/code: such pools can now keep serving text traffic without a client-side rewriting proxy. The config is `image_generation_policy` (`allow`/`strip`/`block`) on the key's limits, edited as a three-way select in the key editor; the legacy `disable_image_generation` boolean maps to `block`, so existing configurations keep their exact behavior. Standalone `/v1/images/*` endpoints and image-only models remain governed by block. The strip applies uniformly across the HTTP, WebSocket, and compact `/v1/responses` paths; the pre-existing WS anti-stall strip (issue #220) now reuses the same, more complete implementation.
- **Payload rules can match the actually-scheduled account — pro keys backfilling into a plus group no longer force-fast their own pro accounts (#410).** The existing `group_ids`/`group_names` identity gates match the account groups a key is *allowed* to use, which is independent of where a given request is actually routed — so a pro key allowed `[pro, plus]` with a tier-override rule hit the gate on every request, tagging pro-account traffic with `service_tier` overrides meant only for the plus fallback. Rules gain three new gates matched against the account actually selected for the current attempt: `account_group_ids`, `account_group_names`, and `account_plans` (plan values as in the key plan filter: free/plus/pro/prolite/team/k12/go; all three wildcard lists with the usual OR-within/AND-across semantics). The rule identity is derived per attempt after scheduling picks the account (a copy — the base identity is shared across attempts), so a retry that switches accounts re-evaluates the gates against the new account, and the usage log's requested-tier attribution flips with it; a rule setting any `account_*` gate fails closed on paths where no account is resolved. The rule editor exposes the three gates in both the template and custom forms, relabels the old gates as "key-allowed groups" to make the two semantics unmistakable, and — the actual trap behind the issue — the "Fast Group Boost" template now writes `account_group_names` instead of `group_names`, matching what its description always claimed ("requests served by accounts in this group"); re-save an existing template rule to adopt the new semantics. Verified live: with the gate set to the serving account's plan the override lands (`requested_tier=priority`), with a non-matching gate it doesn't.

### Fixes

- **`system`-role messages on the native Responses path are rewritten to `developer` — SDKs no longer 400 with "System messages are not allowed" (#409).** The Codex upstream accepts only the `developer` role for system-style messages. The chat-completions and Anthropic translation paths have always converted `system` → `developer`, but the native `/v1/responses` passthrough forwarded `input[]` items verbatim, so any SDK that builds Responses requests with a `{"role":"system"}` message got the upstream's `invalid_request_error` back. The shared body-preparation pipeline now rewrites the role on message items (both the typed `{"type":"message","role":"system"}` form and bare `{"role":"system"}` items), preserving content untouched; it runs after `previous_response_id` cache expansion (so replayed history is covered too) and before content-part normalization, and therefore applies uniformly to the HTTP, WebSocket, and compact entrypoints. The OpenAI-relay path is deliberately left alone — the official Responses API accepts `system`. Verified live against the real upstream on `gpt-5.4-mini` and `gpt-5.6-sol`, streaming and non-streaming, typed and bare forms: requests complete and the system instruction demonstrably steers the reply.

## v2.5.7 - 2026-07-18

### Features

- **Client and upstream user-agent auditing in usage logs (#403, by @Torrekie).** Each request now records both the inbound client user-agent and the outbound user-agent actually sent upstream, threaded through the HTTP and WebSocket-relay executors and the upstream drain path, so operators can see which Codex CLI / Desktop / SDK version a request came from and what UA the proxy presented to OpenAI. The Usage page surfaces both, with PG and SQLite persistence and migrations.
- **Request-prep phase timings for `/v1/responses` (#405).** The stretch between the handler entry and the first upstream attempt is now instrumented and emitted two ways: an `X-Codex2API-Phase-Timing` response header (written before the first SSE byte, so it's visible in a packet capture) and a structured `[TIMING]` log line. Segments are `mw` / `read` / `validate` / `prepare` / `schedule` plus `body_kb`; the existing `first_token_ms` picks up from the attempt onward. This closes the attribution gap where a gateway saw a slow first token but codex2api's `first_token_ms` looked fast. The companion "native fast-path" idea in the issue was measured and dropped — `PrepareResponsesBodyForOwner()`'s full `Unmarshal → map → Marshal` costs only 1–13ms even on 400KB bodies, not worth the regression surface; `validate` is the real prep-phase cost (grows linearly with body size) and is the right target if the prep phase is ever optimized.
- **Self-learning size-based transport router for oversized requests (#404).** Instead of the incremental-WS-continuation the issue proposed (dropped — official Codex CLI packet captures show every turn sends the full input with `store:false` and `previous_response_id:null`, so upstream WS continuation under `store=false` is unverified with counter-evidence, and `response.id` is account-bound which fights pool scheduling/affinity), a `websocketSizeRouter` (`proxy/ws_size_router.go`) records the smallest request-body size seen at each close-1009 (only learning from ≥64KB samples to avoid misclassification and test pollution) and dispatches subsequent requests at ≥95% of that threshold straight to the HTTP upstream — eliminating the "wait for the WS attempt to fail, then serialize HTTP" first-token tail. The learned threshold recalibrates on a 6h TTL; all four 1009 trigger points across both handlers feed the learning. Gated behind a `codex_ws_size_router_enabled` system setting (default on, wired DB→Store→runtime→admin API→Settings UI) with a `CODEX_WS_SIZE_ROUTER=off` escape hatch.

### Fixes

- **Responses Lite protocol marker is preserved across transports — GPT-5.6 requests no longer 400 with `Unknown parameter: 'input[N].namespace'` (#406).** GPT-5.6 (Sol / Terra / Luna) run in Responses Lite, where history items may carry an optional `namespace`. The gateway wasn't carrying the lite signal to the upstream, so the upstream parsed the request as ordinary Responses and rejected the `namespace` field. The HTTP header allowlist now passes `x-openai-internal-codex-responses-lite`, and the WS path writes it into `client_metadata` (`ws_request_header_x_openai_internal_codex_responses_lite`), so the marker survives on both transports (`bc9b78a`). Separately, lite constraints are now normalized before upstream dispatch (`9501dbb`): `reasoning.context` forced to `all_turns` and `parallel_tool_calls` to `false` on both transports, and on the HTTP transport the hosted `image_generation` tool and its bridge text are stripped (WS keeps them so the image bridge can still fall back) — the auto-injected image tool otherwise made every lite request over HTTP 400. Verified live against the real upstream over forced-WS.
- **Payload-rule editor round-trip and `service_tier` override sanitization (#395).** The visual rule editor's edit/delete buttons were `opacity-0` hover-only (invisible on touch) and `formStateFromEntry` silently dropped headers / `not_match` / `exist` / `not_exist` on save; buttons now hide only under `hover:hover` and a `carry` field round-trips the unexposed conditions, with simple rules routed back to their semantic template on edit. The append-prompt / override-prompt / service-tier / effort-map templates now expose the group-name/-id and key-name gates added in #398. And a rule overriding `service_tier` to `flex`/`auto` no longer 400s upstream: handler-layer sanitization (which only lets `priority`/`fast` through) ran *before* payload rules applied, so a rule-injected tier bypassed it — `sanitizeServiceTierForUpstream` now re-runs after rules apply, and the editor's tier input is a whitelist Select.

## v2.5.6 - 2026-07-16

### Features

- **Payload rewrite rules can now target by API key and account group, and usage stats bill the post-rewrite `service_tier` (#395).** The payload rule engine previously matched only on model and inbound headers, so operators who split subscription tiers into account groups and hand out different API keys per group had no way to scope a rule to a key — e.g. "requests using the *fast* key get `service_tier: priority`". Each rule gains four new match gates — `api_key_ids` / `api_key_names` / `group_ids` / `group_names` — all wildcard lists with the same OR-within / AND-across semantics as the existing `models` gate (IDs are matched as stringified values, so `fast*` works on names and `3*` on ids). The request identity (key id/name + the key's allowed account-group ids/names, group names resolved from a new in-memory `Store` cache kept in sync on group create/rename/delete) rides the upstream `context.Context` into `ExecuteRequest` — no signature churn on the WS/relay path — and a rule that sets any identity gate fails closed when the request has no resolved identity (internal / unauthenticated paths). Separately, the usage log now attributes `service_tier` from the body *after* payload rewrite: the four Codex-path handlers (responses / messages / chat-completions / WS) recompute the requested tier via `EffectiveRequestedServiceTier` per attempt before sending, so a rule that overrides `service_tier` is reflected in `RequestedServiceTier` / `BillingServiceTier` (the "requested" billing policy) instead of logging the client's pre-override value. Compact/relay paths keep their existing behavior (they don't apply payload rules). A one-click "fast group" template is added to the rule editor.
- **Self-service account submission portal (#393).** A standalone public page `/account-portal` lets external contributors add their own ChatGPT/Codex OAuth account to the pool without going through an admin: enter a contact email → generate a ChatGPT authorization link → sign in → paste the redirected URL back (the portal parses `code`/`state` client-side, since OpenAI only registers the `localhost:1455` callback that a remote user's browser can't deliver to the server) → the account lands in a **pending-review** state. Submitted accounts are stored with `enabled=false` + a `self-service` tag and are **not** added to the scheduling pool; an admin approves (enable) or rejects (delete) them from a new "pending self-service" review panel on the Accounts page. Approving an account that was never loaded into the runtime pool now pulls it into the scheduler on the spot. The whole flow is gated behind a `public_account_portal_page_enabled` toggle (default off; page and API both 404 when disabled), protected by per-IP rate limiting and email-format validation (no CAPTCHA this round), and never echoes tokens back to the submitter. Accounts also gain a general-purpose `note` field (admin-editable, `PATCH /accounts/:id/note`) that the portal populates with the submitter's contact email as an identifier.

### Fixes

- **Upstream account 403s now rotate to another account instead of being passed straight back to the client (#396).** A Codex upstream 403 is always account-side (`payment_required` / `deactivated_workspace` / `codex_access_restricted` — OAuth / plan / workspace scope, never request-content), and the offending account was already cooled down and hard-excluded on 403 — but 403 was missing from `isRetryableStatus`, so the request stopped immediately and surfaced the raw 403, and clients like Claude Code treated it as their own permission failure and quit, even when the pool still held healthy accounts. 403 is now retryable (bounded by `max_retries`), so the request switches accounts across every endpoint; once retries are exhausted the final error is rewritten to a pool-level `503 account_pool_forbidden` (mirroring the existing 401→503 handling) rather than transparently forwarding the 403. Account cooldown/error semantics for 403 are unchanged.

## v2.5.5 - 2026-07-15

### Features

- **Payload rewrite rules engine, with a dedicated management page (#321/#390/#391).** A declarative rule engine lets the admin rewrite the upstream-bound request body — append or override `instructions` (the system prompt), override `service_tier`, conditionally map reasoning effort (e.g. `medium → high`) — without hanging another rewriting proxy in the chain. Six rule groups (`default`/`default_raw` write-if-missing, `override`/`override_raw` unconditional, `append` string-append, `filter` field-delete) are gated by match conditions that must all hold: model wildcards, header wildcards, JSON-path equality (`match`/`not_match`) and path existence (`exist`/`not_exist`). Rules apply once at the top of `ExecuteRequest` — before the WS/HTTP fork — so all four inbound protocols (responses / chat / anthropic / client WS) are covered uniformly; image-generation requests are skipped to protect the image bridge's constructed `instructions`. A protected-field blacklist (`model`/`stream`/`input`/`store`/`prompt_cache_key`/`type`) rejects rewrites that would desync scheduling, billing, or session isolation (model rewriting stays with the existing model-mapping feature). Rules persist in system settings (PG + SQLite), hot-apply on save with 400 on invalid JSON or protected fields, and `GET /api/admin/settings/observed-instructions` records the most recent client-passthrough `instructions` as a reference sample. The editor lives on a new sidebar page `/payload-rules` with a full visual mode — stat tiles, per-group color-coded rule cards with match-gate chips, one-click templates for the four typical rules, a custom-rule form with a model multi-select (portal-positioned dropdown with search, wildcard input, keyboard navigation) — two-way synced with a JSON source mode. The UI warns that overriding `instructions` replaces the official Codex CLI prompt (which can affect its tool-calling behavior) and invalidates the upstream prompt cache, so rule text should stay stable.
- **Configurable prompt-filter hardening and NewAPI audit integration (#389, by @ifThink404).** The prompt filter gains strict terminal rules, text normalization, composite matching, cumulative risk scoring, input *and* output stream scanning across the chat / responses / Anthropic paths (SSE and WebSocket alike, with correct output buffering and fragmented-event scanning), and a prompt-intelligence history with hardened auto-add validation. A NewAPI-facing audit channel persists configurable audit secrets and policy settings, verifies signed user/IP headers (the V1 signature covers method, path, and a body digest, with replay protection), and enforces repeat-offender bans using per-key cache leases for risk and offense counters. The raw advanced-JSON controls are replaced by a localized admin UI with secret generation/replacement flows, rule comparison/history, a regex preview dialog, in-app usage guides for both Prompt Filter and Payload Rewrite, and Nginx buffering guidance; settings and secret updates are atomic, and strict-keyword false positives are corrected.
- **Public Image Studio portal for API-key users (#369).** A standalone `/image-studio` UI backed by `/api/image-studio` endpoints lets teammates generate and edit images with their own API key, without admin-console access — per-key isolation, a gallery reference picker, and a polished studio layout.
- **Dashboard pool runway (#383).** Turns the existing limit-pressure forecast into an ops-facing runway metric (~N hours / 1 day+ / under 1h) with a dashboard card, action hints, and a show/hide toggle (local preference). Forecast math lives in `frontend/src/lib/poolRunway.ts` (real long-window length, skip absent 5h samples); the accounts recovery panel reuses the same runway copy.

### Fixes

- **Inbound request bodies are decompressed per `Content-Encoding` — zstd bodies from new Codex clients parse correctly (#373/#388).** Codex Desktop/CLI 0.144.2+ under a `requires_openai_auth` + bearer-token configuration compresses the `/v1/responses` body with zstd; the gateway never decompressed inbound bodies, so the compressed bytes were parsed as plaintext JSON — a large body failed local validation with "Field 'model' is required", while a small one could slip past validation on literal residue inside the zstd frame and leak compressed bytes upstream ("The 'None' model is not supported"). A new `RequestBodyDecompressor` middleware (after `RequestSizeLimiter`, before `BodyCacheMiddleware`) decodes zstd / gzip / x-gzip / br / deflate (zlib-wrapped with a raw-flate fallback), unwinds stacked encodings in RFC 9110 reverse order, and rewrites the raw-body cache, `c.Request.Body`, and `Content-Length` while stripping `Content-Encoding` so all downstream validation and forwarding sees plaintext. - **Compaction turns are exempt from the first-token watchdog (#381/#387).** A compaction round must ingest the entire history before emitting its first compaction output item, so its first frame can legitimately exceed a short configured first-token timeout (e.g. 30s) — and because that round's `encrypted_content` is bound to the original account, the timeout-triggered cross-account retry was near-certain to keep failing until attempts were exhausted and the session was dead. `firstTokenTimeoutForRequest` now returns 0 (watchdog off) when the request body carries `compaction_trigger`, wired into the Responses relay and Codex paths plus the inbound WS path; ordinary requests keep the configured threshold, and a genuinely hung turn is still bounded by the client's own timeout.

Stale 5h state is now cleared only after authoritative 7d-only observations, while empty or partial responses preserve existing 5h snapshots and unrelated unauthorized / error / disabled / newer rate-limit states; cooldown and source ordering stay safe across the WHAM, Responses, Anthropic, images, persistence, and runtime-cache paths. Status badges and the account detail sheet name the specific affected window, and the UI hides an absent 5h usage/cost instead of showing misleading placeholders.
- **HTTP/2 upstream connections send keep-alive PINGs so dead connections are evicted.** Direct Codex/OpenAI HTTP connections negotiate HTTP/2 via ALPN, but `http2.Transport` defaults to `ReadIdleTimeout=0` (no keepalive PING), so a connection silently severed by an intermediate proxy/NAT looked alive on both ends — a request landing on it hung until the OS TCP retransmission timeout (minutes), surfacing as extreme first-token latency. WebSocket relays already have full Ping/Pong keepalive and are unaffected.
- **Image-edit reference-image cap relaxed from 10 to 16 (#275).** The 10 was a defensive value from an earlier review pass, unrelated to upstream capability; the official Images API allows 16 reference images for gpt-image edits, and this proxy's responses image channel was verified to accept 16 `input_image` items end-to-end. - **Codex models manifest passthrough + automatic model learning.** Codex CLI and the Codex desktop app refresh their model picker from the provider's `GET /models?client_version=...` (custom provider mode) or `GET /backend-api/codex/models` (chatgpt_base_url mode), expecting the ChatGPT manifest format (`{"models":[{slug,...}]}`). codex2api only served the OpenAI-compatible list, so Codex clients silently fell back to their local cache — the model picker froze on whatever existed the day the provider was configured, and new models never appeared. The manifest is now proxied verbatim (body + ETag, `If-None-Match`/304 supported) using a schedulable ChatGPT OAuth account's credentials: clients always see the account's real, current model entitlements, including capability metadata (`prefer_websockets`, `truncation_policy`, input modalities) that the OpenAI list can't carry. `client_version` acts as the dispatch fingerprint — requests without it keep the OpenAI-style list unchanged. On top of that, unknown slugs in a fetched manifest are learned into the model registry asynchronously (strictly add-only: existing rows — including admin-disabled ones — are never touched, absent models are never deleted since a manifest reflects one account's entitlements), so a brand-new upstream model passes request-side model validation as soon as any Codex client refreshes its picker, with no manual sync or release needed.
- **Continue-thinking: optional auto-continuation for truncated Codex reasoning (#335, #338).** New system settings `codex_continue_thinking_enabled` (default off) and `codex_continue_max_rounds` (default 8, range 1–32). When enabled, the `/v1/responses` streaming path detects the `518n−2` reasoning-token truncation fingerprint, re-sends the current turn's reasoning (ids stripped, `encrypted_content` preserved, tagged as commentary) back upstream on the same account, and folds multiple upstream rounds into a single downstream SSE response — sequence numbers and output indices are renumbered end-to-end, usage is rebuilt from the single-response perspective, and per-round real consumption is logged separately for billing. Disabled, the path is byte-for-byte identical to before.
- **Reasoning effort: `none`/`minimal` pass through, `ultra` pre-wired.** The upstream accepts `none/minimal/low/medium/high/xhigh`, but the proxy's whitelist only knew `low..xhigh` and silently clamped everything else to `high` — quota-saving `none`/`minimal` requests got maxed out instead. Both are now forwarded as-is, and an `ultra` tier is pre-registered (settings, reasoning-effort model aliases, frontend selectors) so an upcoming model tier can be used without a code change; `max`→`xhigh` mapping is unchanged. Usage-log effort badges are color-coded on a cool-to-hot palette across all tiers.
- **Built-in `gpt-5.6-sol`/`gpt-5.6-terra`/`gpt-5.6-luna` fallback.** The gpt-5.6 series (Sol/Terra/Luna) that appeared on the official site is pre-registered in the built-in model list, so it becomes requestable the moment the upstream opens access — marked `APIKeyAuthAvailable=false` like gpt-5.5 (ChatGPT-account models). Pricing is unpublished and bills via the default fallback until official rates land.
- **Output tokens-per-second column in usage logs.** Usage table and mobile cards show output throughput (output tokens ÷ generation seconds). When a first-token time is recorded it is excluded from the window — closer to pure decode speed, so TTFT doesn't drag the rate down; failed or empty-output requests show `-`.
- **Admin UI overhaul (#336).** Mobile navigation gets primary tabs plus a "more" sheet with dense tables converted to cards on small screens; theme settings gain follow-system mode, light/dark swatches, and one-click reset; the accounts page gets clickable KPIs, status chips, pinned row actions, and a floating detail drawer; API keys becomes a product-console surface (filterable KPIs, search/sort, `last_used_at` from usage logs, one-time create reveal, optimistic updates); settings layout unifies control sizes and pins the docs TOC while scrolling.

### Fixes

- **Body-signal compact requests on relay-account pools are promoted to the compact chain.** Newer Codex clients (remote compact v2) embed the session-compaction trigger as an input item (`type: "compaction_trigger"`) in a plain `POST /v1/responses` instead of calling `/v1/responses/compact`. ChatGPT OAuth accounts' native upstream accepts that form directly (verified: returns compaction output items), so passthrough is kept — such requests are pinned to official accounts whenever one is available, preserving the native SSE response shape. But a relay (OpenAI Responses API) account's plain `/v1/responses` generally rejects the trigger or returns a non-compaction response, leaving the client stuck with "expected exactly one compaction output item, got 0"; when no official account is available (relay-only pools), the request is now promoted at handler entry to the `ResponsesCompact` chain, hitting the relay upstream's `/v1/responses/compact` — the compact chain's own `stream` stripping ensures the JSON response isn't misparsed by streaming logic into account-rotation retries.
- **Namespace-style `image_gen` tool declarations no longer get the hosted image tool stacked on top.** Codex's /image skill declares image generation as `{type:"namespace", name:"image_gen"}` (top-level `tools[]` or nested in Responses Lite `input[].additional_tools`), which the auto-injection logic didn't recognize — so those requests got a hosted `image_generation` tool *and* bridge instructions injected on top, and the bridge text explicitly assumes the `image_gen` namespace is absent: duplicated tooling with self-contradicting instructions. Both injection decision points (ChatGPT and OpenAI relay paths) now detect the namespace form and pass such requests through untouched.
- **Asymmetric `arguments` types on replayed input items no longer 400 (#330).** The upstream requires `function_call.arguments` to be a JSON *string* but `tool_search_call.arguments` to be an *object*; clients or the `previous_response_id` cache replaying a previous turn's output items verbatim tripped "Invalid type for 'input[N].arguments'" — only in conversations that had used tool search. A normalizer now fixes both directions after cache expansion; the input-type whitelist also gains upstream-supported types (`shell_call`, `apply_patch_call`, `mcp_call`, …) so the proxy doesn't reject legal replays before the upstream sees them, and the call-id continuation cache covers `shell_call`/`apply_patch_call`.
- **Multi-line upstream WS error frames are no longer truncated in SSE.** An upstream error frame's `error` object can be pretty-printed JSON with real newlines; embedding it verbatim into a single-line `data:` SSE event cut everything after the first newline — logs, DB, and clients all saw just `..."status_code":400,"error":{`. Error objects are now compacted to one line (`json.Compact`), with a final safety-net compaction before WS→SSE writes.
- **Removed the upstream HTTP client's 10-minute total timeout (#287).** `http.Client.Timeout` spans the entire request lifecycle including body streaming, so long streaming turns (complex tasks can exceed 10 minutes) were cut off mid-transfer. The standard transport now matches the uTLS path's staged control: `Timeout: 0` with lifecycle owned by the request context (downstream disconnect cancels), a new 5-minute `ResponseHeaderTimeout` to catch upstreams that never answer, and unchanged dial timeout and stream-stall detection.

### Build

- **Go builder image bumped to `golang:1.26.5-alpine`.** Container builds pick up the 1.26.5 stdlib security fixes.

## v2.4.6 - 2026-07-08

### Features

- **Configurable retry interval and sticky same-account retry for transport errors (#331).** Two new system settings refine retry behavior; defaults preserve the old behavior exactly. `retry_interval_ms` (0–30000, default 0 = immediate) inserts a wait between retry attempts — covering transport errors, upstream non-200 retries, pre-first-token stream-break transparent retries, and WS silent retries across all proxy paths — and aborts immediately if the client disconnects mid-wait (the account's concurrency slot is released before sleeping). First-token-timeout retries skip the extra wait since they already burned a full timeout. `transport_retry_policy` (`rotate` default / `sticky`) changes what happens on connection-level failures (network blips, proxy node switches): in sticky mode the request retries on the *same* account — no hard exclusion, session affinity kept, and no `ReportRequestFailure`, so a local network hiccup no longer burns through the whole account pool while marking every account unhealthy. Upstream HTTP status errors (5xx/401/429) still rotate, since those are per-account verdicts. Both settings live next to "max retries" in the admin settings page.

### Fixes

- **Streaming requests that fail before the first token now return real HTTP errors instead of 200 (#318, reported and first-fixed by @JrCx7scC).** When the upstream terminated a stream with `response.failed` (e.g. `context_length_exceeded`) before any token, codex2api wrapped the failure into a 200 SSE stream (plus `[DONE]` on the chat path), so billing middle layers treating codex2api as an upstream saw a normal-looking success with no usage and billed their users by *estimated input tokens* — "upstream rejected with 0 output, user billed for the full input". The original PR's direction was right but didn't survive the wire: the post-loop flush committed the 200 header before the error JSON could be sent (gin's `Flush()` calls `WriteHeaderNow()` even on an empty buffer), so the fix additionally guards the epilogue flush on "wrote any body", overrides the pre-set SSE Content-Type, and gates the error return on an explicit abort flag. Applied across Responses/ChatCompletions streaming, the OpenAI Responses direct branch, and inbound WebSocket — where there is no HTTP status to fix, so non-retryable pre-first-token failures now send a structured `error` frame and close with a semantic close code (429→1013, other 4xx→1008, 5xx→1011) instead of relaying the raw failure frame and ending the turn as if it succeeded. Retryable failures keep the existing silent-retry/relay contract.
- **wham 401 no longer unilaterally bans an account — the `/responses` probe is the final arbiter (#328).** codex_at (bare AT import) accounts can be permanently 401 on the wham endpoint (token missing a workspace claim) while real `/responses` traffic works fine; the usage probe treated that 401 as dead credentials and put the account into a 24h unauthorized cooldown — re-banning it after every manual reset, or flapping banned/unbanned every probe round when fallback probing was on. A wham 401 is now a sentinel: no failure report, no ban. When fallback is available the `/responses` probe delivers the auth verdict (200 recovers / 401 bans); in wham-only mode the 401 is just logged, leaving genuinely dead tokens to be cooled down by real-traffic 401s.
- **Duplicate-account merge race could soft-delete both accounts.** After a merge, the async probe reload on the surviving account could trigger a reverse merge before the losing row was soft-deleted — the reverse pass merged old→new, then the outer pass soft-deleted the new row too, losing the account entirely (reproduced deterministically in CI where the fake probe returns instantly). Merges are now serialized under a mutex and the losing row is soft-deleted before the probe reload, so concurrent or recursive merges always see each other's results.
- **Online update hardening (#327, #329).** After an in-place update the restart re-exec could still launch the *old* binary — `os.Executable()` resolves `/proc/self/exe`, which after the atomic swap points at the deleted old inode; restart now execs the explicit new binary path. The frontend no longer caches the whole update-info blob (a stale `has_update=true` survived the update and kept showing "new version available" when current == latest); only the remote release fields are cached and the comparison is recomputed against the running bundle version on every check. The update/restart UI also gains client-side timeouts and a "refresh manually" hint when the restarted process doesn't answer within 90s.
- **Usage stats "source account" column shows email again.** A v2.4.5 frontend change made the account *name* take precedence, so unnamed AT-imported accounts all rendered as their placeholder name (`at-account-1`) instead of their email identity; the order is restored to email → name → ID on both the usage and operations views.

### Security

- **Bumped `quic-go` to v0.59.1 and `golang.org/x/crypto` to v0.52.0.** The quic-go bump fixes GO-2026-5676 (HTTP/3 QPACK trailer-expansion memory exhaustion), flagged by govulncheck as reachable through the actual call path; the x/crypto bump clears a Dependabot alert.

## v2.4.5 - 2026-07-07

### Features

- **Per-account model mapping for OpenAI Responses accounts (#325).** Each OpenAI Responses (relay API-key) account can now carry its own model alias map, so different relay channels can expose different model names while routing to the same upstream — e.g. mapping a channel-specific alias onto the upstream's real model name per account, on top of the existing global model mapping. Follow-up hardening: mapping keys and values both run through `ValidateModelName` (length + charset) before being accepted, since mapped source aliases flow into `/v1/models` responses, model validation, and usage logs — arbitrary strings are rejected instead of injected.
- **Connection-test content supports multi-line random pick and variable templates (#320).** Building on #317's configurable single test sentence, the test content setting now accepts multiple lines: each probe randomly picks one non-empty line, and `{{...}}` placeholders in the picked line are expanded before sending — `{{time}}`/`{{date}}`/`{{datetime}}`/`{{timestamp}}`/`{{rand}}`/`{{rand:min-max}}` (min>max auto-swaps; unknown variables are kept verbatim). This removes the "every account in a deployment sends the same probe sentence" fingerprint. Single-line configs behave exactly as before; all three test entrances plus the usage probe share the same rendering.
- **Reset-credit expiry details in the account usage dialog (#322).** The "manual quota reset" section now fetches the per-credit breakdown from the upstream's zero-cost reset-credits list endpoint and shows each credit's expiry time, sorted soonest-first with credits expiring within 7 days highlighted — so you can see not just how many resets are left but when each one becomes worthless, and burn the soonest-expiring ones first. The authoritative available count from the detail endpoint also recalibrates the locally cached number. Detail fetch failures degrade gracefully: the count and the reset button keep working, only the expiry list shows an "unavailable" hint.
- **Content-derived session seeds: multiple users behind one shared API key now stick to accounts per-conversation.** When a single API key is used as a channel for a whole downstream site, sessionless requests all collapsed onto the same key-derived affinity key — every user squeezed onto one upstream account and account switching was all-or-nothing per key. A content-derived seed (sha256 over conversation-stable fields: model + instructions + tools + system/developer + first user message) now slots in between explicit session signals and the per-key fallback, so each conversation converges onto its own sticky account and the pool load spreads out. Requests continuing via `previous_response_id` skip the seed (their input changes every turn and would drift), the seed only participates in local routing (never sent upstream in the default isolation mode), and the session-binding table gains a 65536-entry soft cap with expired-entry sweeping.

### Fixes

- **Forced-WS + shared single API key no longer bleeds context across users.** With `codex_force_websocket` on and one API key shared by multiple end users, user 2's turn could get stitched onto user 1's conversation: the reused stateless connection's handshake headers (`Session_id`/`Conversation_id`) were frozen to the *first* request's cache key and never updated, so an upstream that keeps per-connection conversation state saw every later user under the first user's session identity (same family as #268/#308). Stateless reused connections now send *no* session identity in the handshake at all — per-request isolation rides entirely on the per-request `prompt_cache_key` in the frame body; explicit-session and per-api-key modes are unchanged. A `CODEX_WS_STATELESS_ONESHOT=1` escape hatch disables slot reuse entirely (one connection per request, hard isolation, at the cost of per-request handshakes). Continuation affinity is also fixed: `previous_response_id` context lives only inside the connection that produced the response, so continuation requests now return to their original connection (response→connection bindings with TTL and a 4096 cap, guarded by account + API-key ownership + connection-pointer checks) instead of intermittently hitting "previous response not found" on a random slot.
- **`previous_response_id` context cache is isolated per downstream API key.** The response cache was keyed globally by `response_id` and expansion never checked ownership, so any request presenting (guessing, or accidentally reusing) someone else's response id would get that user's full conversation items injected into its input — a cross-key context leak. Cache entries are now namespaced by owner (`key:<id>`/anon) across all three write/expand paths (HTTP streaming + non-streaming, WS-native ingress, translator inline expansion); an owner mismatch is treated as a plain cache miss.
- **DB failures and upstream account 401s are no longer misreported as client 401s (#323).** Two layers of "server/account-side trouble" surfaced to clients as "your API key is invalid": (1) API-key resolution collapsed "key not found" and "DB failure" (connection exhaustion, timeouts) into the same result, so under load a PG "too many clients" turned into 401 `invalid_api_key` — resolution is now three-state, real DB errors return 503 and skip the `AUTH_FAILED` audit log so credential-attack alerting stays clean; (2) an upstream account's revoked OAuth token, after retries were exhausted, passed through as a raw 401 — non-`missing_scope` upstream 401s are now rewritten to a 503 pool-level error (`account_pool_unauthorized`), clearly distinct from client auth failure. Account-side handling was already correct (401 accounts get banned + cooled down); this fixes only the status-code leak to clients.
- **Free/team monthly-window accounts label their long usage window "30d" instead of "7d" (#324).** v2.4.4 taught the backend to recognize monthly (~30 day) rate-limit windows, but the frontend still hard-coded the long-window label as "7d" — misleading for free and team accounts whose long window really is a month. The account list usage bars, billed column, and scheduler board badges now render the real window length (`30d` for monthly, actual day count for other lengths, `7d` fallback when unknown). Also fixes the 7d billed-cost window start, which was hard-coded to `reset − 7 days`: for a monthly window that start date landed in the future, so `billed_7d` was stuck at 0.

## v2.4.4 - 2026-07-05

### Features

- **One-click online update from the admin console (#319).** The version popover's "new version" hint gains an "Update now" button: the backend (`GET`/`POST /api/admin/system/update`) picks the release asset matching the running OS/arch, downloads it from GitHub only (https allowlist re-validated on every redirect hop, 200MB cap), verifies SHA256 via the release asset digest or `SHA256SUMS.txt`, atomically swaps the binary (keeping a `.backup` for manual rollback, restoring the old binary if the swap fails mid-way), and re-execs in place on Unix; the frontend then polls until the new version answers and reloads. Dev/non-semver builds and Windows are marked unsupported instead of pretending to update, and release/Docker builds now embed the version via ldflags so the backend reports it accurately. Follow-up: container environments (docker/podman/k8s, detected via `/.dockerenv`, `/run/.containerenv`, `/proc/1/cgroup`) get an explicit warning that an in-container update is reverted when the container is recreated and image pull is the right upgrade path — the update itself stays available for temporary hot-fixes.
- **Configurable connection-test content (#317).** The prompt sent by single/batch/recycle-bin connection tests and the usage/plan-sync fallback probes was hard-coded to `"hi"` — an easily fingerprintable traffic signature shared by every codex2api deployment. It is now a system setting (Settings → test content, auto-saved, defaulting to `"hi"`), applied uniformly across all five test/probe paths.
- **Per-account custom upstream headers (#316).** Each account can carry a set of custom headers (create/edit/import/scheduler flows all supported) that are applied to outgoing upstream HTTP and WebSocket requests after the standard headers, so they can also override them — the primary use case being `Chatgpt-Account-Id` to pin a multi-workspace team login onto a specific workspace. Header names/values are validated (RFC token charset, no CR/LF, 64 headers / 8192-char values max) and stored in the existing credentials JSON (no migration); accounts without custom headers behave exactly as before. Follow-up fix: the wham usage/reset-credit probes and Codex invites now use the *effective* workspace id (custom-header override first) instead of the OAuth-decoded one, so when a header reroutes traffic to another workspace, the quota bars, auto-pause, and smart pacing measure the workspace the traffic actually lands on; and wham identity write-back is skipped for overridden accounts so the override can't pollute the OAuth dedup identity.

### Fixes

- **Team monthly (30-day) windows are no longer misclassified as 7d — smart pacing stops over-throttling team accounts.** The usage-window classifier only had 5h/7d slots, but a team plan's second rate-limit window is actually monthly (~2592000s), and it was being stuffed into the 7d slot. Smart pacing computed its natural burn rate against a hard-coded 7 days, making the allowed pace ~4.3× too strict for monthly windows and needlessly throttling team accounts. Windows within a 28–31 day tolerance are now recognized as monthly (`IsMonthlyWindowSeconds` as the single source of truth, on both the wham and passive-header paths), the real window length feeds the pacing math, and the account list labels the window "30d" instead of "7d". Known limit: the window length lives in memory only, so between a restart and the first probe (seconds) pacing falls back to the 7d default.
- **Rate-limit cooldowns and 5h/7d window resets trigger an immediate wham probe when their countdown hits zero.** Usage probes were driven only by the periodic background sweep (default 2 min), so when a cooldown or window reset expired, the account flipped back to "available" instantly but its usage bar kept showing the stale pre-reset value until the next sweep. A boundary timer now stays armed on the earliest upcoming reset/cooldown boundary across all accounts and fires a zero-cost wham probe the moment it passes (with a 2s lag for clock skew), re-arming for the next boundary; cooldown/reset writers wake it lock-safely, and the periodic sweep remains as a fallback.

### Features

- **Import accounts by pasting a full ChatGPT Session JSON (#314, #315).** The Add Account dialog gains a "ChatGPT Session" tab and the Import dialog gains a "paste text" button, so you can paste the raw JSON from `chatgpt.com/api/auth/session` and import in one step instead of manually splitting out `accessToken`/`email`. - **Optional smart pacing spreads remaining quota evenly across a usage window (issue #312).** When enabled, if an account is burning a 5h/7d window faster than an even pace (used% running ahead of elapsed-time%), its dynamic concurrency is scaled down to a sustainable rate so the remaining quota lasts until the window reset instead of hitting the limit early. Pacing is per-account (each account paces on its own window, so the pool is smoothed as a whole), reuses the existing guard-band concurrency machinery (no new background task — it rides the usage snapshots already being refreshed), takes the stricter of the 5h/7d ratios, and never drops below a configurable floor. ### Fixes

- **Credit accounts skip 5h/7d usage-window rate limiting (#313).** `credit_enabled + credit_skip_usage_window` accounts are meant to bypass usage-window limits, but response-header sync and the WHAM probe still marked them `rate_limited` whenever a 5h/7d window hit 100%, so a credit account was excluded from local scheduling despite the flag. A shared `SkipsUsageWindowLimits()` gate now records the usage snapshot but skips the local rate-limit marking (`MarkUsage7dRateLimited` / premium 5h) for such accounts, while real cooldown/error paths (401, existing cooldowns, genuine upstream failures) are preserved.
- **codex_at accounts merge into an existing same-identity account after the identity probe; import progress separates new vs updated.** codex_at tokens (`at-...`) carry no decodable JWT, so their OAuth identity is unknown at insert and only backfilled later by the WHAM probe — the post-probe merge the RT path already performs was missing here, so re-adding a codex_at could leave a duplicate. Manual codex_at adds now trigger a probe, and once it backfills email+account_id the importer re-checks for a same-identity account and merges into it (soft-deleting the new row, preserving usage stats). Separately, import/add now counts an "updated" bucket distinct from "new": hitting an existing identity only refreshes its credentials and no longer counts as a new add (fixing the success+duplicate double-count where re-importing the same accounts showed a nonzero "new"), and a single pasted access token now streams a progress bar too.
- **A 5h/7d window reset now triggers an immediate verification probe.** After a window's reset time passes an account looks "available" again, but it may actually be banned upstream — and the post-reset refresh was gated behind the usage `maxAge`, leaving up to ~10 minutes in which a bad account kept getting scheduled and failing. `NeedsUsageProbe` now fires a probe promptly when the reset boundary is crossed and the snapshot predates it (no `maxAge` delay), probing once per boundary; a real 401 then marks the account unauthorized.
- **A WebSocket upstream `close 1008 (policy violation)` now triggers async auth verification.** On the forced-WS transport a token that is actually invalid is rejected by the upstream as a `close 1008` rather than an HTTP 401, so it was classified as a generic transport failure — the account only lost health points, was never cooled down, and kept being scheduled and failing (the real `Unauthorized` was visible only on the HTTP transport). A WS upstream close now asynchronously runs a zero-cost WHAM probe for that account: a real 401 marks it `unauthorized` (matching the HTTP path), while a content-policy/transient 1008 leaves a healthy account untouched. Throttled to once per 30s per account.

## v2.4.1 - 2026-07-03

### Features

- **Bare RT imports now auto-merge into an existing same-identity account, preserving usage stats.** Scenario: an account was first imported via AT, then its bare RT was imported later. A bare RT carries no identity (it is not a JWT), so import-time dedup necessarily lets it through, and the identity only becomes known after the first refresh exchanges it for an AT — previously the flow stopped there, leaving two accounts (AT + RT) for the same identity in the pool. After a successful refresh the importer now re-checks for an identity duplicate: on a hit, the new credentials are merged into the existing account (writing the refresh token upgrades it to auto-renewable — RT takes highest priority) and the newly inserted row is soft-deleted. The merge only updates credential/identity keys — `codex_*` usage snapshots are untouched and the surviving account keeps its ID, so usage statistics and per-account request history are fully preserved. Copies imported with "allow duplicate add" are skipped.
- **AT batch add gains a progress bar.** `POST /accounts/at?stream=true` streams per-token SSE progress (aligned with the RT/ST path); the frontend automatically switches to streaming when more than one AT is pasted.

### Fixes

- **AT imports dedup by `user_id` identity — duplicate accounts no longer pile up.** Two stacked holes let the same account enter the pool repeatedly: (1) a personal-plan AT's JWT may carry no workspace `chatgpt_account_id`, only a `user_id` — identity dedup required email+account_id, so it fell back to raw access-token comparison, and since ATs rotate, a re-imported newer AT never matched the stored one; (2) the wham probe used to backfill `user_id` into the `account_id` field when the workspace id was missing, polluting the dedup key so even workspace-bearing JWTs never matched. JWT parsing now extracts `user_id` (with `chatgpt_user_id` fallback), the identity key is email + (account_id or user_id), the wham backfill no longer writes user_id into account_id, and `FindActiveAccountByOAuthIdentity` matches the `user_id` key as well as legacy polluted `account_id` values. Covers both the add dialog and TXT/JSON file imports.
- **Data migration v2 merges existing user_id-shaped duplicate accounts on startup.** The identity aliases now include the `user_id` key, and the dedupe migration reruns once to merge stock duplicates (both correctly-keyed `user_id` rows and rows whose `account_id` was polluted by the old wham backfill) — no manual cleanup needed; losers are soft-deleted into the recycle bin with audit events.
- **Copies imported via "allow duplicate add" are exempt from identity dedup and migration merges.** Forced duplicates (e.g. the same account with different proxies) were indistinguishable from accidental ones, so the dedupe migration would have merged away copies the user kept on purpose. Forced imports are now tagged `allow_duplicate=true` in credentials across all four add/import paths; the dedupe migrations skip tagged rows, and identity matching skips them too so a later normal import updates the primary account instead of writing new credentials into a forced copy. Note: only copies imported after this version carry the tag — force-imported duplicates from v2.3.9/v2.4.0 will still be merged by the v2 migration.
- **`context_length_exceeded` no longer triggers account-rotation retries (#310).** When the upstream returned `context_length_exceeded` as a `response.failed` frame (no explicit status code), the error-code string matcher didn't recognize it, fell through to the default 500 classification, and the request was transparently retried across up to 3 accounts — each attempt necessarily failing again while charging an innocent account a server-failure health penalty.
